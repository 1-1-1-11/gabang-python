# gobang-python 产品化升级任务计划

## 0. 已确认范围

本计划用于把当前项目从“本地演示项目”升级为具备现代前端 UI、AI 算法优化、自动化验收、独立审查和 GitHub 留痕机制的真实可持续项目。

已确认要求：

- 先忽略打包相关任务。
- 前端使用 Vue3 + Vite，目标是精美 UI 和可维护组件结构；这是用户新确认的产品化方向，不代表恢复或迁移旧 JS/React 项目源码。
- AI 算法优化需要实际改动，不只写文档。
- 不按天数衡量任务，按模块、阶段和验收门槛推进。
- 每个任务测试、验收、subagent 独立审查通过后，允许自动 commit 并 push 到远程 GitHub。
- subagent 独立审查结果保存到 `docs/collaboration/reviews/`。
- 任务要拆到最小模块，每个模块都有执行标准和验收标准。
- 每次工作需要在任务台账、审查文件、commit 和 push 中留痕。
- 用户要求使用 superpowers/ralph skills 制定拆分任务；当前环境中可用的是 `using-superpowers` skill，未提供名为 `ralph` 的 skill。本计划按最小模块化原则拆分，并落实执行标准、验收标准、审查留痕、commit/push 留痕。

## 1. 产品目标拆解

### 1.1 总目标

交付一个现代化五子棋 AI 项目：前端具备精美 UI 和清晰交互，后端 AI 搜索速度和棋力可量化优化，开发过程可验收、可复盘、可追溯。

### 1.2 阶段目标

1. 建立协作与验收机制
   - 仓库内存在统一任务计划文档。
   - 每个任务都有执行标准、验收标准、测试命令、subagent 审查要求和 GitHub 留痕规则。

2. AI 算法优化
   - 先建立 benchmark 和战术测试基线。
   - 再优化局部终局判断、候选点生成和走法排序。
   - 每次优化都有指标对比，避免只凭体感判断。

3. Vue3 + Vite 前端现代化
   - 建立现代前端工程结构。
   - 将游戏页面组件化。
   - 做出精美棋盘、控制面板、AI 信息区、落子记录和错误/结束状态。

4. 质量与复盘体系
   - 每个最小任务完成后自测。
   - 每个任务必须由 subagent 只读独立审查。
   - 验收通过后 commit 并 push。
   - 台账记录任务状态、测试结果、审查结论、commit hash 和 push 状态。

## 2. 当前条件分析

## 3. 关键难点

| 难点 | 风险 | 处理方式 |
| --- | --- | --- |
| 前端重构范围大 | 一次性重写导致 E2E 大面积失效 | 先搭 Vue3 + Vite 架构，再逐个模块迁移 |
| AI 优化可能降低棋力 | 候选点局部化或排序变化漏防 | 先补战术测试，再改算法 |
| subagent 审查不聚焦 | 审查结果不可执行 | 每次审查必须带任务 ID、diff 范围和验收标准 |
| 每任务 commit/push 增加流程 | 开发节奏变慢 | 任务拆小，每个 commit 只做一件事 |
| 引入 Node/Vite 后运行方式变化 | README、E2E、脚本不一致 | 前端初始化任务必须同步更新 npm scripts 和 E2E 启动方式 |
| 计划与台账不同步 | 后续 agent 误领已完成任务或重复实现 | 每个任务完成后同时更新本计划的复盘区和 `TASKS.md` 状态 |
| 文档要求与实际工程边界冲突 | Vue/Vite 任务被误解为恢复旧 React/JS 项目 | 只允许新建 Vue3 + Vite 代码，不复制旧源码、旧静态资源或旧构建配置 |
| AI 指标被偶然波动误导 | 单次 benchmark 无法证明优化有效 | 固定局面、重复运行、同时记录 nodes/beta_cutoffs/cache_hits/耗时和选点正确性 |

### 阶段 B：AI 优化基线与第一轮算法优化

验收门槛：

- AI benchmark 覆盖开局、中盘、立即胜、必须防、简单算杀。
- 战术测试断言关键坐标，而不是只断言合法落点。
- 至少完成一个低风险性能优化模块。
- 后端相关测试通过。
- 记录优化前后同一批局面的指标对比，至少包含 `nodes`、`beta_cutoffs`、`cache_hits`、耗时和最终选点；历史记录中的 `prunes` 对应当前 `beta_cutoffs`。
- subagent 审查无 Blocker/Major。
- commit 并 push。

### 阶段 C：Vue3 + Vite 前端架构初始化

验收门槛：

- Vue3 + Vite 工程可安装、启动、构建。
- API 客户端模块存在。
- 游戏状态管理模块存在。
- E2E 或基础浏览器验收可访问新前端。
- README、CLAUDE 或启动脚本中的前端运行说明与 Vite 方式一致。
- subagent 审查无 Blocker/Major。
- commit 并 push。

### 阶段 D：精美 UI 与交互体验

验收门槛：

- 棋盘、棋子、控制区、状态区、落子记录、AI 信息区完成组件化。
- 支持难度选择、AI 思考状态、按钮禁用、错误提示、游戏结束提示。
- Playwright 主路径通过。
- 浏览器实际验收通过。
- 桌面端和移动端关键布局无重叠、无按钮文字溢出、无棋盘变形。
- subagent 审查无 Blocker/Major。
- commit 并 push。

### 阶段 E：复盘、质量收敛与留痕

验收门槛：

- 后端全量测试通过。
- 前端构建和 E2E 通过。
- AI benchmark 有优化前后对比。
- 任务台账完整记录 commit、push、审查文件和测试结果。

## 6. 每个任务的通用完成流程

每个最小任务必须按以下顺序执行：

## 9. MVP 定义

本轮 MVP 以“现代前端 + AI 优化 + 可验收协作流程”为核心。

1. `docs/collaboration/任务计划.md` 落地。

- 公网部署。
- 用户登录。
- 房间系统。
- 排行榜。
- 比赛系统。

## 17. B-10 节点/时间预算预研

### 17.1 当前实现边界

- 请求侧只有 `depth`：`StartGameRequest.depth` 与 `MoveRequest.depth` 限制在 `1..8`，`play_ai_move()` 将它传给 `minmax()`。
- 响应侧只有 `score`、`best_path`、`current_depth`，没有 `nodes`、`prunes`、`cache_hits`、`elapsed_ms`、`limited` 或 `limit_reason`。
- 搜索侧已有全局 `search_metrics`，但 API 没有把它封装进 `GameSnapshot`；这使前端 D-08 的搜索信息展示还缺后端稳定字段。
- `minmax()` 目前是同步深度搜索，没有 deadline、节点上限、取消标记或迭代加深；强行加入时间预算会改变“返回完整深度结果”的语义。

### 17.2 决策

本轮暂缓直接实现节点/时间预算参数，原因如下：

- API 合同影响较大：新增请求字段和响应字段会触发 `schemas.py`、`game.py`、OpenAPI、README、前端 skeleton 测试和未来 Vue API client 的同步修改。
- 当前搜索不是迭代加深；时间到即停会面临“没有完整候选结果”或“返回上一个完整深度结果”的设计选择，不能只在递归里插一个 `deadline` 判断。
- 阶段 B 已完成一轮可量化优化，B-10 的目标是给出可执行下一步，不应阻塞 C-02 前端架构初始化。

### 17.3 后续可执行方案

建议把预算能力拆成两个后续最小任务：

1. C-09：搜索指标响应字段
   - 修改范围：`backend/app/schemas.py`、`backend/app/game.py`、`backend/tests/test_api_contract.py`、`backend/tests/test_game_api.py`、README。
   - 设计：新增 `SearchMetricsSnapshot`，在 `GameSnapshot` 中返回 `nodes`、`prunes`、`cache_hits`、`cache_stores`、`candidate_moves`、`leaf_nodes`、`elapsed_ms`。
   - 验收：现有 API 兼容；未搜索时指标为空或零值有明确语义；D-08 可直接展示。

2. 节点预算优先于时间预算
   - 修改范围：`backend/app/minmax.py`、`backend/app/game.py`、`backend/app/schemas.py`、后端 AI 测试和 API 测试。
   - 设计：先引入 `node_limit`，用确定性节点计数停止搜索；响应返回 `limited=true` 和 `limit_reason="node_limit"`。
   - 验收：`node_limit` 下返回合法 move；立即胜、必须防守、连续威胁在默认无预算路径不退化；预算路径有单独测试。

时间预算建议在节点预算稳定后再做：

- 使用 `time.monotonic()` deadline。
- 优先配合迭代加深，返回上一个完整深度的最佳结果。
- 响应字段明确 `limit_reason="time_limit"` 与实际 `elapsed_ms`。

### 17.4 当前结论

- B-10 不新增后端代码。
- 预算参数暂不进入阶段 B。
