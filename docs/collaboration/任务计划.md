# gobang-python 产品化升级任务计划

## 0. 已确认范围

本计划用于把当前项目从“本地演示项目”升级为具备现代前端 UI、AI 算法优化、自动化验收、独立审查和 GitHub 留痕机制的真实可持续项目。

已确认要求：

- 先忽略打包相关任务。
- 前端使用 Vue3 + Vite，目标是精美 UI 和可维护组件结构；这是用户新确认的产品化方向，不代表恢复或迁移旧 JS/React 项目源码。
- AI 算法优化需要实际改动，不只写文档。
- 不按天数衡量任务，按模块、阶段和验收门槛推进。
- 每个任务测试、验收、subagent 独立审查通过后，允许自动 commit 并 push 到远程 GitHub。
- subagent 独立审查结果保存到 `docs/collaboration/reviews/`。
- 任务要拆到最小模块，每个模块都有执行标准和验收标准。
- 每次工作需要在任务台账、审查文件、commit 和 push 中留痕。
- 用户要求使用 `/superpowers` skill 制定拆分任务；当前环境中该 skill 名称不可用。本计划按最小模块化原则拆分，并落实执行标准、验收标准、审查留痕、commit/push 留痕。

## 1. 产品目标拆解

### 1.1 总目标

交付一个现代化五子棋 AI 项目：前端具备精美 UI 和清晰交互，后端 AI 搜索速度和棋力可量化优化，开发过程可验收、可复盘、可追溯。

### 1.2 阶段目标

1. 建立协作与验收机制
   - 仓库内存在统一任务计划文档。
   - 每个任务都有执行标准、验收标准、测试命令、subagent 审查要求和 GitHub 留痕规则。

2. AI 算法优化
   - 先建立 benchmark 和战术测试基线。
   - 再优化局部终局判断、候选点生成和走法排序。
   - 每次优化都有指标对比，避免只凭体感判断。

3. Vue3 + Vite 前端现代化
   - 建立现代前端工程结构。
   - 将游戏页面组件化。
   - 做出精美棋盘、控制面板、AI 信息区、落子记录和错误/结束状态。

4. 质量与复盘体系
   - 每个最小任务完成后自测。
   - 每个任务必须由 subagent 只读独立审查。
   - 验收通过后 commit 并 push。
   - 台账记录任务状态、测试结果、审查结论、commit hash 和 push 状态。

## 2. 当前条件分析

### 2.1 后端条件

已有：

- `backend/app/main.py`：FastAPI 游戏 API。
- `backend/app/game.py`：游戏会话管理。
- `backend/app/board.py`：棋盘规则、落子、悔棋、胜负判断和评估入口。
- `backend/app/evaluation.py`：棋形识别、候选点评分和候选点生成。
- `backend/app/minmax.py`：negamax + alpha-beta 搜索、VCT/VCF 入口和搜索指标。
- `backend/tests/`：后端测试覆盖。

不足：

- `Board.get_winner()` 当前以全盘扫描为主。
- `Evaluate._point_groups()` 和 `_role_score()` 会反复扫描全盘空点。
- 搜索缓存是简单 exact tuple，没有 bound 语义。
- 缺少时间预算、节点预算和迭代加深。
- benchmark 偏 smoke，不足以支撑算法优化复盘。

### 2.2 前端条件

已有：

- `frontend/index.html`、`frontend/styles.css`、`frontend/app.js` 可完成基本对局。
- 前端已接入 start/move/undo/end API。
- Playwright E2E 已覆盖基础浏览器路径。

不足：

- 当前原生静态前端偏 demo。
- 组件边界不清晰。
- UI 视觉完成度不够产品化。
- 难度选择、AI 思考反馈、结束状态、复盘体验仍需增强。

### 2.3 协作条件

已有：

- `docs/collaboration/TASKS.md` 作为任务台账。
- `docs/collaboration/reviews/` 存放历史审查记录。
- `docs/collaboration/CLAUDE_REVIEW_PROMPT.md` 已定义审查输出格式和门禁。

需要补齐：

- 新阶段详细计划入口。
- 每个任务的统一完成流程。
- 每个任务的审查留痕和 GitHub 留痕规则。

## 3. 关键难点

| 难点 | 风险 | 处理方式 |
| --- | --- | --- |
| 前端重构范围大 | 一次性重写导致 E2E 大面积失效 | 先搭 Vue3 + Vite 架构，再逐个模块迁移 |
| AI 优化可能降低棋力 | 候选点局部化或排序变化漏防 | 先补战术测试，再改算法 |
| subagent 审查不聚焦 | 审查结果不可执行 | 每次审查必须带任务 ID、diff 范围和验收标准 |
| 每任务 commit/push 增加流程 | 开发节奏变慢 | 任务拆小，每个 commit 只做一件事 |
| 引入 Node/Vite 后运行方式变化 | README、E2E、脚本不一致 | 前端初始化任务必须同步更新 npm scripts 和 E2E 启动方式 |

## 6. 每个任务的通用完成流程

每个最小任务必须按以下顺序执行：

1. 领取任务
   - 在 `docs/collaboration/TASKS.md` 标记任务为进行中。
   - 确认任务 ID、目标、主要文件、执行标准、验收标准。

2. 实现任务
   - 只修改任务范围内必要文件。
   - 不混入无关重构。
   - 不引入旧 JS/React 项目源码。

3. 自测
   - 后端任务至少运行相关 pytest。
   - 前端任务至少运行 npm build 或相关 E2E。
   - AI 任务必须运行 AI 搜索和 benchmark 相关测试。
   - 文档任务至少检查 git diff 和文档链接有效性。

4. subagent 独立审查
   - 使用只读审查 subagent。
   - 审查 prompt 必须包含任务 ID、diff 范围、执行标准、验收标准。
   - 审查输出保存到 `docs/collaboration/reviews/`。
   - 存在 Blocker/Major 时必须修复后再次审查。

5. 更新留痕
   - 更新任务状态。
   - 记录测试命令和结果。
   - 记录审查文件路径和结论。

6. commit + push
   - 验收通过后必须在用户已授权范围内创建新 commit 并 push 到远程 GitHub。
   - commit 只包含当前任务相关变更。
   - 台账记录 commit hash 和 push 状态。

## 8. subagent 审查模板

审查文件保存路径：`docs/collaboration/reviews/<任务ID>-<任务名>.md`

审查输出格式：

```markdown
# Review: <任务ID>-<任务名>

## Scope

- 本次审查范围。

## Blocker

- 无，或列出必须修复后才能验收的问题。

## Major

- 无，或列出高风险缺陷、接口不一致、测试缺口。

## Minor

- 无，或列出可后续处理的小问题。

## Question

- 无，或列出需要实现方确认的问题。

## Verdict

- PASS / FAIL。
```

合并和推送门禁：

- 存在 Blocker 或 Major 时，不允许 commit/push 作为已完成任务。
- Minor 和 Question 可以记录后通过，但必须在台账说明处理决定。

## 9. MVP 定义

本轮 MVP 以“现代前端 + AI 优化 + 可验收协作流程”为核心。

MVP 必须包含：

1. `docs/collaboration/任务计划.md` 落地。
2. Vue3 + Vite 前端工程初始化。
3. 新前端能完成开始、落子、AI 回复、悔棋、结束。
4. UI 比当前 demo 明显更精美，具备组件化结构。
5. AI 至少完成一轮可量化优化，benchmark 有前后对比。
6. 后端相关测试通过。
7. 前端构建或 E2E 通过。
8. 每个完成任务都有 subagent 审查记录。
9. 每个完成任务都有 commit 并 push 到 GitHub。
10. 任务台账能追溯任务状态、测试结果、审查结论和 commit。

MVP 不包含：

- 打包 zip。
- 公网部署。
- 用户登录。
- 房间系统。
- 排行榜。
- 比赛系统。
- 神经网络 AI。
